## 3. 依存関係と推奨シーケンス

```
   ②香受けMC（強さ・独立）  ──►  v98   ← Sonar指示の主眼
   ①memo空間削減（挙動不変） ──►  v99   （③の前提を作る）
                    │
                    ▼
   ③昇格パラメータ再検討 ──────►  v100  （①完了後にのみ着手）
```

- **v98 = ②香攻めへの受けMC評価**（本決定）。①③と独立に実施でき、プロジェクト目標（人間相方との協力を支援する強さ）に直結する強さ改善のため、先行が妥当。
- **v99 = ①memo空間コスト削減**。挙動不変・低リスクで、③のブロックを外すインフラ整備。
- **v100 = ③昇格パラメータ再検討**。①完了後、OOMリスクが解消してから資源昇格の是非をA/Bで判断。

補足（代替順序）: もし「③昇格を最速で解禁する」ことを最優先するなら ①→③ を先行し②を後回しにする順序もあり得る。今回は Sonar が強さ改善（②）を優先する判断のため、上記シーケンスを推奨とする。

## 4. プロジェクト固有の順守事項（全候補共通・goita-dev-loopより）

- **フラグ伝播の罠**: gated A/B（ランタイムフラグ）は使わない。source-levelのエンジン変種を無条件化して mixed-engine head-to-head（paired standard error）で直接対戦する（②③）。
- **byte-exact規律**: 単一ファイルHTMLのため、意図しない差分混入を sha256（正準規約＝波括弧深度抽出・`)();`は含めない）で機械確認。`saveCareerToStorage`/`loadCareerFromStorage` は不変を維持。
- **デプロイ基準**: 性能・挙動に関わる変更は「勝率が悪化していない（winrate-neutral以上）」をA/Bで確認してからデプロイ（②③）。①は決定等価が同等以上の保証。
- **ゴールの確認**: 目標は自己対戦勝率の最大化ではなく、**人間パートナーとの協力を支援できる強さ**。②のレビューでは「勝率は上がったが協力プレイの観点で悪化していないか」を必ず観点に含める。
