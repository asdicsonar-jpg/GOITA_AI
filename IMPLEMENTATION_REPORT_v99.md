# IMPLEMENTATION_REPORT_v99 — v99「memoコンパクト格納」実装報告

- 担当: goita-dev-loop フェーズ2(Executor)。ROADMAP① の**値側レバー**(v98 の a/b/c に続く増分)。
- 目的: v98 で未達だった **<300MB**(促成モードのピークRSS)を達成する。v98 は約420MB止まりで、残メモリの支配要因は「memoの per-entry JSオブジェクトオーバーヘッド」だった(実測: 700kエントリの `Map<string,float>` = +383MB、うちキー文字23BはごくわずかでオブジェクトヘッダとMapスロットが支配)。
- **性質: 純粋な性能最適化(挙動変更なし)。** 受け入れの主柱は決定等価の機械証明。

## 1. 変更(G/G_B 同一diff)
`ddRootValues` の置換表を、**大探索(nodeCap≥1,000,000=促成/振り返り)に限り** `Map` から **open-addressing の typed-array**(`Uint8Array`キー＋`Float64Array`値＋`Uint8Array`占有)へ置換。
- per-entry の JSオブジェクトオーバーヘッド(~547B/entry)を除去。実測700kエントリで **383MB→45MB(~8.5×)**。
- キーは v98(c) で全 Latin1(≤255)化済みゆえ `Uint8Array` に**無損失**格納。
- 容量 `C=nextPow2(nodeCap*1.4)`、エントリ数 ≤ nodeCap(budget上限:`ddSearch`毎に`--budget.n`・setは非cut時のみ)ゆえ**負荷率≤0.71で溢れなし**。決定化ごとは占有ビット`fill(0)`で再利用(v98 lever b と同義=同時生存1本)。
- **標準(小探索 nodeCap<1e6)は高速な `Map` を継続**。理由: コンパクトの占有クリア(`fill(0)`=O(C))が小探索では割高で、標準テンポを 271ms→433ms/局に悪化させたため、**しきい値で経路分離**して標準経路の回帰を回避した。
