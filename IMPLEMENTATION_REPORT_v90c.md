# 実装レポート v90c「定着」

- Executor: Sonnet5
- 対象計画: `PLAN_v90_teacher.md`（改訂3版）v90c セクション（S-1〜S-3）
- 基準ファイル: v90b納品版（`goita-app-v90b.zip`、index.html 16,212行・`CACHE_NAME="goita-v98"`）
- 成果物: index.html 16,404行・`CACHE_NAME="goita-v99"`

## 1. 実施した変更

### S-1: SRSストア（`goita_srs_v1`）＋保存優先ポリシー
`SRS_STEPS=[1,3,7,17,40]`日、`SRS_MAX_CARDS=100`、`SRS_BUDGET_BYTES=256KB`。`srsGrade(card, ok)` は計画のコード例をそのまま採用（正解でease+1・最大4、失敗でease=0・lapses+1、dueを再計算）。

**保存優先ポリシー（欠陥4対処）**: `saveCareerToStorage()` は本コミットで一切変更・呼び出ししていません（byte-exact確認済み、下記4節）。SRS側の保存（`srsSaveStore()`）は`goita_career_v1`に触れる経路を一切持たない自己完結の関数として実装し、予算・件数超過時は「定着済み（ease降順）→失敗少（lapses昇順）→古い（last昇順）」の順に間引き、`setItem`失敗（quota等）時は半減して最大3回再試行、それでも失敗すれば`goita_srs_v1`を削除して黙って諦めます。

### S-2: `srsSlim`による局面のカード化
`srsSlim(rec, upto)`は計画のコード例に準拠し、`parent`/`initialHands`/`upto`/`moves(seat+actionのみ)`だけを残します（`why`/`cf`/`thinkMs`/`scoresBefore`/`matchNo`/`gameNo`等は保存しません）。

- **詰め**: 生成時点（`runPuzzleGen`内、解答前）の`rec`/`idx`から`srsSlim`で凍結スナップショットを作り`drillMeta.srsPayload`に保持します（解答中に伸び続ける`app.rec.moves`ではなく、生成時点の不変なスナップショットを使用）。正解が確定した瞬間（即時一致・結末判定の両経路とも）に`srsAddPuzzle()`で登録します。復習中（`d.srsCard`が設定されている場合）は新規登録ではなく既存カードの`srsGrade()`を行い、**既存の`app.puzzleStats`は二重に加算しません**（新規詰めの解答時は従来どおり加算）。
- **定石**: `openJoseki()`の`#jk-try`クリック時に`{key}`のみを`srsAddJoseki()`で登録。同じkeyの重複登録はしません。
- **復習の判定方式**: `rebuildState()`で凍結局面を再現し、`G.freshSolverProbe()`で読みを取り直して判定します。`generatePuzzle()`は一切呼びません。
- **型（PATTERNS）は対象外**です。計画どおりv90fへ分離します。

### S-3: 期日の稽古札への合流
`keikoProposeDue()`を実装し、`#keiko-panel`の優先度1（v90a T-1で確保済みの枠）に接続しました。期日超過カードの件数と最も古いカードの経過日数を事実として表示します（例:「11日前に覚えた詰め・定石が3問あります。まだ解けますか？」）。「復習する」で`srsReviewStart()`が最大5枚のキューを組み、詰めは既存のdrill判定パイプラインを再利用、定石は`openJoseki()`に「思い出せた／忘れていた」の自己申告2択を追加して判定します。「今日はいい」は既存の`goita_keiko_v1`当日抑制を共用します。表示件数自体は事実として制限していません（「残り」という圧力表現のみを避けています）。

## 2. 計画からの逸脱

1. **SRS復習時の詰め判定は「初手一致」のみで行い、有段（band="adv"）の「上がり切り」判定は再現しません。** `srsSlim`は計画の指定どおり`band`を保持しないため、復習時にどの難度帯だったか区別できません。全ての復習詰めを基本・中級と同じ初手一致判定として扱っています。既存のadv判定ロジック（M1・v79）自体は変更していません。
2. **定石の復習UIは既存の`openJoseki()`モーダルに自己申告2択ボタンを追加する形で実装しました。** 計画のS-3ワイヤーフレームは「復習する」ボタンまでを規定していますが、複数枚（詰め＋定石混在）のキュー処理・各カードの判定方法（詰めはSolver再計算、定石は自己申告）は計画に詳細記述が無いため実装判断で補いました。
3. **期日カードの稽古札文言は「n日前に覚えた〜」を、キュー内の最も経過日数が長いカードを基準に算出しています。** 混在種別（詰め・定石）を1つの文言に集約するための実装判断です。

## 3. 検証結果

- **単体動作確認（jsdom, 8ファイル・全PASS）**: `test_s1.js`（SRS_STEPS遷移・削除順・破損データフォールバック）／`test_s1_priority.js`（quota注入でも`goita_career_v1`が無傷であることと、SRS操作がcareerキーに一切書き込まないこと）／`test_s2.js`（凍結局面の盤面ハッシュ一致・srsSlimが不要フィールドを持たないこと）／`test_s2_hooks.js`（新規詰め正解での凍結登録+puzzleStats加算、SRS復習中はpuzzleStats不加算+重複カード非生成、定石の重複登録防止）／`test_s3.js`（0枚非表示・1枚以上表示・「残り」非表示・セッション最大5枚）／`test_s_quota.js`（100枚の実測: 123,967B≈121.1KB、予算256KBの47.3%、1枚平均1,240B）／`test_s_forbidden.js`（due札・定石復習画面の禁止語彙0件）。
- **回帰確認**: 既存の16ファイル（v90a: t0〜t5・smoke・ongame・settings_hook・misc_hooks／v90b: g1・g2_propose・g2_trial・g2_stats・g2_wording・g3）を再実行し全PASS。v90bまでの機能への回帰なし。
- **byte-exact確認**: `G`/`G_B`エンジンIIFEをv90b基準とSHA-256一致で確認（両者完全一致）。加えて`saveCareerToStorage()`/`loadCareerFromStorage()`本体もv90b基準と文字列完全一致を確認（S-1受け入れ基準3）。
- **axe-core構造的検証**: keiko-panelの「due」カード（openCareer()の卒業帳と同時表示）・定石SRS復習画面（自己申告2択付きopenJoseki()）を実際にレンダリングした状態で4テーマにわたりスキャン。`region`（moderate）は18ノード（v90a/v90b基準の19ノードより1少ない — 理由: due札はgrad_propose札と異なり`#keiko-diag`を非表示にするため、可視ノードが1つ減るのみで新規カテゴリの追加はありません）。`aria-dialog-name`（serious, 1ノード）はv90bで報告済みの`modal()`共通仕様に起因するもので変化なし。定石SRS復習画面の独立スキャンでも新規カテゴリはゼロでした。
- **禁止事項の機械確認**: due札・定石復習画面を含む出力DOM（script/style除去後）に対し、指定11語彙に加え「残り」「サボ」も含めてgrepし、4テーマ・独立スキャンとも**0件**を確認。

## 4. 既知の懸念点

1. **有段（band="adv"）詰めのSRS復習時判定**: 上記「計画からの逸脱」1に記載のとおり、初手一致判定に統一しています。有段の「上がり切り必須」という難度特性が復習時には再現されないため、有段の詰めをSRS登録した場合、復習の体感難度が生成時より下がる可能性があります。
2. **定石の復習判定は完全な自己申告制**です（Solverのような客観判定を持ちません）。「思い出せた」を押せば無条件でease+1されるため、厳密な想起検証にはなっていません。
3. **R-c5（実機でのquota算定）は未検証**です。100枚時の実測値（121.1KB、合成データ）は報告しましたが、実ブラウザでのUTF-8/UTF-16 quota算定差・実データでの分布は本セッションでは測定できません。
4. **v89安定10テストの再構築**は引き続き未対応です（v90a報告からの継続懸案）。
