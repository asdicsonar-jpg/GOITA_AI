## 0. 再現fixtureのデータについて（計画からの逸脱・要確認）

計画の「付記」は「再現fixture用の報告局棋譜はSonar提供のログ（第4局・親:西、配牌...、#1〜#47）をそのまま用いること」とし、本計画書の末尾に棋譜（#1〜#47の手順）が付属している前提で書かれていましたが、**アップロードされた計画書ファイルには実際の手順一覧（#1〜#47）が含まれていませんでした**（配牌と冒頭の状況説明のみ）。

そのため、test (a) の再現fixtureは、報告された局面の**論理的な構造**（親:西、相方(北)が香でリード、南が唯一の香保持者として後の独立した攻め番で香を攻める、攻め通し番号が香の窓≤8以内）を可能な限り忠実に再現する形で、実際のエンジン（`G.newGameState`/`G.advance`）を使って一手ずつ合法性を確認しながら独自に構築しました。以下で確認済みです:

- 修正前のコード（`goita-app-v90.2.zip`のUI層をそのまま）にこのfixtureを通すと、実際に`{type:"kyo2", honest:false, text:"...裏付けがありません。相方が期待してズレます。"}`という偽宣言判定が再現されることを確認しました（本レポート§3に詳細）。
- 修正後は`{type:"kakari_gotae", honest:true}`に変わることを確認しました。

**もし報告局の実際の47手の棋譜と厳密に一致させたレビューが必要であれば、その棋譜データをご提供いただければ差し替えて再検証します。** 今回のfixtureは「同じ根本原因（相方リード→本人の独立した応え攻め）を持つ、機構的に等価な再現」であり、修正の正しさを示す証拠として十分と判断していますが、この判断自体もご確認をお願いします。

## 1. 実施した変更

### 変更1: detectSignal()に応え文脈ガードを追加

既存の「同席再攻めガード」（`st.history.some(h => h.seat === seat && ...)`）の直後に、計画のコード片をそのまま追加:

```js
if (st.history.some(h => h.seat === G.partnerOf(seat) && h.act === "attack" && h.koma === k)) return null;
```

相方が既にどこかで同駒を攻めていれば（リード限定ではなく、非リードへの追随も含め）新たな符丁宣言とは判定しない。エンジン読み側の`kakariGotae()`のled規則（v90.1のリード限定修正）と対称の線引き。

### 変更2: computeSignals()にかかり応えの肯定的認識を追加

`detectSignal()`呼び出しの**前**に、`G.kakariGotae({seat: mv.seat, hand: st.hands[mv.seat], hist: st.history})`を使った判定を追加。該当すれば`mv.sig = {type:"kakari_gotae", honest:true, jk:"kakari", text:"..."}`を設定して以降の処理をスキップする。

**実装時の是正（計画からの軽微な逸脱）**: 計画のtext案は文中に「（かかり応え）」という括弧書きを含んでいましたが、`sigHtml()`は`mv.sig.jk`（今回は`"kakari"`）から`JOSEKI.kakari.name`（＝「かかり応え」）を自動的に文末へチップとして追記する既存の仕組みを持っています。計画のtext案のまま実装すると「…正しい符丁です（かかり応え）。（かかり応え）」という二重表示になってしまうため、text側から「（かかり応え）」の括弧書きを除去しました。これは前回のv90.2 REVIEW対応で修正した`showYakuSplash`の「組組」二重付与と同種の問題であり、既存の`kgb_pair`/`kyo2`の`text`実装（joseki名を`text`に書かず`jk`側の自動チップに任せる）と一貫させています。

1. **報告局の実棋譜（#1〜#47）が未提供のため、再現fixtureは機構的に等価な独自構築です。** §0に記載のとおり、修正前後の挙動差分（kyo2偽宣言→kakari_gotae肯定表示）は実証済みですが、実際に報告された局面と完全一致するものではありません。
