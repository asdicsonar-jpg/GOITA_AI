# デザイン憲章 (DESIGN_CHARTER)

- 初版: v89a（E-5・PLAN_v89_uiux.md）
- 目的: `index.html` 内にこれまで暗黙のうちに実践されてきた規律を成文化し、以後の実装ループ（`goita-dev-loop` スキル）における Planner/Reviewer の判断基準として使う。本書は既存コードの実例を根拠として引用する。新しい判断が必要になった場合は、まずここに準じ、逸脱するなら理由をレビュー時に明記する。

---

## 1. 色彩憲法

- **真鍮（`--kin`）は「親」と「手番」だけに使う。** 他の意味（強調全般・装飾）に流用しない。
  例: `#statusline.myturn{color:var(--kinari);border-color:rgba(194,163,93,.45)}`（L626）— 自分の手番のときだけ真鍮系の縁取りで強調する。おすすめ駒のグロー（`.koma.rec`）も同じ真鍮系だが、これは「打つべき手」という手番文脈の延長として許容されている。
- **朱（`--shu`）は「立っている攻め」と「上がり」だけに使う。** `#wincut`（上がり演出）・`.fkoma.now`（今まさに攻めの駒）が朱を使う唯一の対象で、他の警告色・強調色として朱を新設しない。
- **青緑（`--aoi`）はリンク・情報表示専用。** `a{color:var(--aoi);...}`（L385）や `#equity-bar .lb-ns{background:var(--aoi)}`（L605、南北＝自陣の形勢表示）がその実例。v70のaxe-core監査で、既定リンク色のコントラスト比が1.8しかなくAA未達だったため`--aoi`に是正した経緯（L280-383付近のコメント）があり、この色は「4.5:1を確保できる情報色」という役割を背負っている。新機能でこの意味を薄める（例: 単なる装飾に転用する）のは禁止。
- **新規UI要素にこの3色を新しい意味で使わない。** 新色が必要な場合は憲章の改訂として提起し、既存3色の意味を上書きしない。
- **検証義務**: 新規/変更した配色は、夜藍（既定）・生成り（`body.theme-kinari`）・高コントラストの3テーマに加え、CUD（`body.cud`、色覚多様性対応の斜線パターン併用。L607-610が実例）と `forced-colors`（Windowsハイコントラスト、L520が実例）の双方で確認する。

## 2. 素材と光

- **駒は実物写真準拠。** v68で「実物の竹駒写真を参考に質感修正」（L295の build コメント）、v80で「駒テクスチャの写真準拠化」（L124）が行われた経緯があり、駒の色・質感を変更する際は写実性を離れない。
  `--koma-back0:#dcaa5e; --koma-back1:#b07f34;`（L326、L3(v80)コメント「裏面を実物写真準拠の飴色の竹皮へ」）、`--koma-bury0:#e7d5a4; --koma-bury1:#cdac70;`（L327-328、v68「実物の竹駒を参考にした伏せ駒用の素地色」）が実例。
- **光源は盤上方やや手前で一貫させる。** 駒の陰影・グロー系エフェクト（`.koma.win-k`のdrop-shadow、`.fkoma.now`のfilter）は上方光源を前提にした`drop-shadow`の向きで統一されている。新規の光効果もこの向きを踏襲する。
- **noto意匠（能登の祭り意匠・キリコ等）は「対局の可読性に重ねない」。** キリコ演出は`#kiriko-fire`として盤外・独立レイヤーで完結し、手駒・河・スコア表示の可読性を阻害する位置には出さない（`z-index`とレイアウトの分離が既存原則）。新規のnoto系装飾も、対局進行に必要な情報の上に重ねない。

## 3. 動きの文法

- **E-1トークンの使用義務。** `:root` に定義した `--ease-glide`（盤上の移動・シート・FLIP、L340）・`--ease-pop`（出現のオーバーシュート、L341）・`--ease-soft`（呼吸・明滅ループ、L342）・`--dur-tap`/`--dur-move`/`--dur-reveal` を新規アニメーションでは必ず使う。トークン外の直値カーブを新設する場合は、既存のどのトークンとも視覚的に区別が必要な理由をレビューで説明する（ホワイトリスト方式。現状の直値残存とその理由は `IMPLEMENTATION_REPORT.md` のE-1節を参照）。
- **reduced-motion全停止は例外なし。** 既存コードには `@media (prefers-reduced-motion:reduce)` によるアニメーション無効化規則が33箇所ある（v89a時点）。新規アニメは必ずこの無効化規則（CSS）またはJS側の `_flipReduced()` ガードを伴わせる。
- **演出は「完了ボタンで待つ」か「スキップ可能」のいずれかであること。** v80のL1原則「wcInは登場+保持のみ。退場は.outで行い、完了ボタンでの持続表示を可能にする」（L926のコメント）が原型。以後の全ての非対話的演出（キリコ・大漁旗・五し相談の間など）は、ユーザーが操作を待たされる場合は必ず完了ボタンを備えるか、タップ等で即座にスキップできる手段を持つ。両方を持たない「見るしかない演出」は禁止。
- **状態の所在を示す要素は瞬間移動させず、物体として滑らせる。** v89c B-1（AIの打牌スライド）で駒の出現を「名札カードから滑ってくる」物理に統一したのに続き、v90「番札」（`#turn-token`）も手番席の移動を`transform`の`transition`で滑らせ、`display`の瞬間切替や座標の瞬間ジャンプ（旧`#turn-marker`のような）を使わない。「今どこに何があるか」を示す要素は、目で追える一つの物体として移動させることを原則とする。

## 4. 音の文法

- **既定OFFを維持する。** `cfg.sound: false`（L9066のcfg初期値）が既定であり、この既定を変更しない。音を追加する新機能は必ず `if (!app.cfg.sound) return;` のガードで開始する（`blip`/`bamboo`/`clack` 等、既存の全音源関数が踏襲するパターン）。
- **合成音は全て100ms級の短音とする。** 既存の`bamboo()`/`clack()`はいずれもオシレーターの減衰を30〜70ms程度で完了させ（例: `g.gain.exponentialRampToValueAtTime(0.0001, t + 0.06)`）、長い持続音を使わない。
- **強度=情報。** 駒の格（点数）と音の重さの対応（低いピッチ・強いゲイン＝重い駒）は、B-4（打音の強弱）以降に導入される規律であり、一度対応関係を作ったら崩さない。

## 5. 情報の節度

- **同時に光ってよいのは「手番」と「最新の攻め」のみ。** `.pcard.pondering`（思案中の手番強調）と `.fkoma.now`（最新の攻め）が現行で許されている常時点滅要素であり、これ以外の要素を同時に明滅させない。
- **無限明滅は使わない。** `.koma.win-k`（勝者の河の駒グロー）は「2回明滅で必ず停止」という明文コメント（L972「無限明滅はC7 v74の教訓で禁止」）を持つ。C7(v74)の教訓（「終局後は最新の攻めの無限明滅を残さない」L9363相当）を踏まえ、新規のバッジ・グロー・明滅は原則有限回とする。
- **バッジ・グロー・明滅の新設は、この節度との交換条件（何を止めるか）を明記してレビューに出す。** 「光る要素」を増やす提案は、既存のどれかを削るか、発火条件を厳密に絞るかをセットで提示すること。
- **手番の表現に朱を使わない。** v90で、手番位置を示していた`#turn-marker`が朱（`var(--shu)`）で描画されていたため「朱=攻めと上がり」の意味論と衝突していた問題を是正し、朱の`#turn-marker`を撤去して真鍮の「番札」（`#turn-token`）に一本化した。手番＝真鍮、攻め＝朱、という対応をどの新機能でも崩さない。

### 5.1 改訂（v90d・R-2「符丁が通じた瞬間」）

- **交換条件付きで例外を1件追加する。** R-2は、人間の符丁的な手（`detectSignal()`で判定・裏付けのある=`signalTruth()`が真のもののみ追跡）に相方AIが整合する応答を返した瞬間、相方の名札カード（`.pcard`）に`--aoi`（情報色。真鍮・朱は使わない）で**有限2回**の明滅（`.pcard.sig-echo`、CSS `animation-iteration-count:2`）を許可する。
- **交換条件（対価）**: この演出が発火している間、**同一カードの`.pcard.pondering`（思案中の明滅）を抑制する**（`step()`側で`!card.classList.contains("sig-echo")`をガードに追加）。相方の名札カードで2種類の明滅が同時に起きることはない。「同時に光ってよいのは『手番』と『最新の攻め』のみ」という本節の原則そのものは変更せず、`.pcard.pondering`と`.pcard.sig-echo`が同一カード上で排他的に扱われることで、実質的な同時発火を防ぐ。
- 既定OFF（`cfg.sigEcho`）。`prefers-reduced-motion`で完全停止（CSS `@media`ガード、既存の`.koma.win-k`と同型）。学習の足場（`SCAFFOLDS`）には含めない — これは既に打った手への事後のフィードバックであり、判断を代行しないため。

## 6. 文化的正確性

- **実在情報（保存会・大会・祭り）は出典確認の上で記載する。** 「宇出津とごいたモーダル」（能登ごいた保存会・大会年6回・あばれ祭等、L288付近のbuildコメント）は実在情報+公式リンクを伴う。M4の番付機能は「アプリ内限定・実在の保存会番付とは無関係」と明記されている（L268、L12103のコメント、L14515のUI文言）— 実在と架空を混同させない実装パターンを踏襲する。
- **監修のない能登弁は使わない。** v72の時点で「N3(能登弁)は保存会等の監修が得られていないため引き続き未実装(意図的)」（L254）と明記されており、この判断は継続する。監修が得られない限り、能登弁の台詞・掛け声を新規追加しない（既存のあばれ祭の囃子詞「イヤサカヨッセ」はN2(v71)で出典確認済みの実在の掛け声であり例外的に許可されている）。
- **実在の船名・団体名を演出に使わない。** v83で大漁旗演出に架空の船名「ごいた丸」を採用し、「実在の船名・団体名は使用しない」と明記された（L95）。以後の演出でも実在の固有名詞（船名・団体名）は使わず、必要なら架空名を使う。

### 6.1 改訂（v90e・D-2「出場前チェックリスト」）

- **保存会・協会が配布する公式ルールPDF等の著作物の内容を、アプリ内に複製しない。** `https://goita.jp/rule/` の遊び方説明書には二次利用・頒布・販売またはそれに準ずる行為を禁止する旨が明記されている。ルールの説明が必要な場合はアプリ独自の表現で書き、原典へはリンクで送る。
- **また、支部・大会・番付など保存会の管轄下にあり増減しうる実在情報を、アプリ内に静的なリストとして列挙しない。** アプリ内の静的リストは必ず陳腐化し、誤った実在情報の掲示になる。公式ページへリンクする（D-2実装例: 支部一覧は `http://goita.jp/ngps/` へ、大会参加方法は `https://goita.jp/taikai/taikaisankahoho/` へリンクするのみとし、支部名や大会名をアプリ内に列挙しない）。
- **是正実例（B-3・v91b）**: 「宇出津とごいたモーダル」（`openUshitsuInfo`）が本節の制定前から残していた「保存会は東京・大阪・神奈川・長野・宮城・金沢・埼玉・福岡など全国に支部を持ち…」という都道府県の静的列挙を、「保存会は全国に支部を持ち…（支部一覧は公式サイトへ）」に丸め、`https://goita.jp/ngps/` への支部一覧リンクを同モーダルに追加した。既存の静的列挙を発見した場合はD-2と同じ方式（丸め文+公式リンク）で是正する。

## 7. 検証規律

- **byte-exact diff**: 計画で指定した行以外に差分がないことを機械diffで確認する。特にエンジン領域（`const G = (() => {...})();` / `const G_B = (() => {...})();`、v89a時点でおおよそL2070-5534 / L5553-9028）はUI改修では一切変更しない。
- **axe-core AA**: 新規トグル・select等の追加後は3テーマ（夜藍・生成り・高コントラスト）で再監査する。
- **Playwrightスモーク**: 起動→対局→振り返り→設定、の一連でconsoleエラーゼロを確認する（具体的なシナリオはPLAN文書の「横断検証・リリース手順」節を参照）。
- **性能変更はA/B**: エンジンの挙動に関わる変更（本憲章の対象外だが将来のため明記）は、フルに無条件化したmixed-engine head-to-headでの検証を要する。**フラグ伝播の罠**（gated A/Bテストはフラグの伝播漏れで結果が歪みうる。「検証したつもりが検証になっていない」パターン）を常に疑うこと。この教訓は `goita-dev-loop` スキルにも明記されている。
