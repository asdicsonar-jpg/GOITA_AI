# PLAN_v98_kyo_receive_mc — v98「香攻めへの受けMC評価」: 相方の香攻めを受けMC評価の対象に含める

- Planner: goita-dev-loop フェーズ1（Fable5相当）
- 発端: v96 build コメント L69-70（F-1のスコープ除外）— 受けMCゲートを **し攻めのみ** に限定し、**香は「相方の香連打計画を乱すリスク」があり観測証拠もないため対象外**として次サイクル候補に記録。deal#128 の「し版」失敗（相方のし攻めを受けず敵に手番を奪われ逆転）と同型の失敗が **香版** でも起きていないかを是正する。
- 基準ファイル: v97承認版 `build/v97/src/index.html`（CACHE_NAME `goita-v117`）。実装対象は `build/v98/src/index.html`（v97のbyte-exactコピーから開始）。
- **性質: 挙動変更（AIの強さ・協力の改善）** — 受け入れの主柱は winrate A/B（mixed-engine head-to-head）。ただし v96 の教訓（F-1''＝もっともらしい閾値でもA/Bで有意悪化）を踏まえ、**診断ファースト**（実データで有害/有益を切り分けてから最小条件を設計）で進める。

## 変更の目的

相方が香で攻めた局面（`isPartnerAtk && attack==="香"`）で、現状のAIは既定でパスする（`receiveChoice` の相方分岐末尾 `{koma: null, why: "味方の攻め — 手番を奪わない（定石）"}`、G: L5268 / G_B: L9316付近）。これが敵に香の手番を奪われ逆転される局面で損をしていないかを、**受けMC評価（`mcDecideReceive`）の対象に条件付きで含める**ことで是正する。同時に、**相方の香連打（きゃんれんだ）計画を乱さない**ことを両立する（v96が香を除外した唯一の理由）。

判断の非対称性（ドメイン知識）:
- パス側の失敗＝敵が香を受けて手番を奪い逆転（deal#128し版の香アナログ）。**重い**。
- 受け側の失敗＝相方の香連打を中断しリードを失う。**これも重い**（← し版には無い、香固有のリスク）。

し版（受け＝軽いテンポ損）と違い、香版は**両側の失敗が重い**。したがって「迷ったら受け」（v96''' のし版タイブレークの発想）を安易に転用できない。**実データでどちらの失敗が優勢かを局面条件別に測ってから**条件を割る。

## 実装方針

診断ファーストの3段階。段階0の結果が「是正不要」なら段階1以降を実装せずクローズする（v96 F-1'' の轍を踏まない）。

### 段階0: 診断採掘（コード変更前・レポート掲載）
1. 800配牌以上（v96診断と同規模）を標準strong optsで自己対戦し、`isPartnerAtk && attack==="香"` かつ **現方策=パス**（"手番を奪わない"）の決定点を全収集。
2. 各点で反実仮想: 「香を受けた場合」に転換して完局まで再生し、転換なしと比較して **beneficial / harmful / neutral** に分類。
3. **香連打の帰趨を必ず区別して集計**: 受けが「進行中の相方香連打を中断した」ケースと「敵の手番奪取を防いだ」ケースを分けてカウント。
4. 局面条件別の判別力を測る（v96''' の (a)&(c) 導出と同じ手法）。候補条件:
   - (a) 応答者の香枚数（受け後も何か継続できるか）
   - (b) 香連打の進行状態（`kyanRendaInProgress`・相方香攻め回数）
   - (c) 敵の最小手駒数 ≤ 4（逆転の切迫度）
   - (d) 未見香（敵が香を受け返せるか＝手番奪取リスクの有無）
   - (e) 敵香集中（`enemyKyoConcentratedStrict`）・香温存が既に発火する局面か
5. **出力**: beneficial保持を維持したまま harmful保持を有意に絞れる最小条件の組。得られなければ「partner-香-pass は有害でない＝是正不要」と結論しクローズ。

### 段階1: ゲート拡張（最小）＋香連打ガード
段階0で有害部分集合が存在した場合のみ。ゲートに香を**香連打非進行を条件に**追加:
```js
(RULE_LEVEL === 0 ||
 (((!isPartnerAtk || attack === "し") ||
   (KYO_RECV_MC && isPartnerAtk && attack === "香" && !kyanRendaInProgress(view.hist, seat)))
  && view.hand.length <= 6))
```
→ 相方香攻めのうち**香連打が進行していない**ものだけを `mcDecideReceive` に回す。香連打進行中は従来どおりパス（相方の計画を保護）。

### 段階2: 香タイブレーク（MC互角時・診断由来の条件）
`mcDecideReceive` が `null`（パス優位/互角）を返したとき、し版タイブレーク（L5599）に**並置**で香版を追加。段階0で確定した条件（例: `margin < 0.10 && 未見香 >= 1 && 応答者香 >= X && 敵最小手駒 <= Y && !kyanRendaInProgress`）を満たすときのみ受け側を選ぶ。閾値 `margin<0.10` は既存の「互角」定数の再利用（新設しない）。

### 案の比較（採用理由）
- **案A（ゲート全開放・香連打ガードなし）**: partner-香を無条件にMCへ。**却下**。MCロールアウト（`rollout`/`policyAction`, L5647）が香連打継続を正しく打てないと pass の価値を系統的に過小評価し、受け過剰バイアス→香連打破壊。
- **案B（ゲート＋香連打ガード＋条件付タイブレーク）= 採用**。v96 F-1（ゲート拡張）＋F-1'''（データ由来タイブレーク）の香版。両側の失敗が重い香の性質に合わせ、香連打を構造的に保護しつつ、敵手番奪取リスクが優勢な部分集合だけ受けへ倒す。
- **案C（タイブレークのみ・ゲート据え置き）**: **却下**。ゲート外では `mcDecideReceive` 自体が走らず `r.means` が得られないため、タイブレークが発火し得ない。ゲート拡張は必須。
- **資源昇格は行わない**（v96 F-1'のOOM・撤回を踏襲）。コストゼロのゲート＋タイブレークに限定。昇格は候補③（v100・memo削減後）へ分離。

## 想定される副作用・リスク

1. **香連打の破壊（最重要・香固有）**: 受けが相方の香連打を中断しリードを失う。→ `kyanRendaInProgress` ガードで進行中を除外＋段階0診断で harmful率を実測。受け入れ基準3で機械証明。
2. **MCロールアウトのバイアス（検証したつもりの罠）**: `rollout` の相方/敵モデルが香連打を継続しないと、pass（相方に続けさせる）側のEVが過小評価され、MCが受けへ偏る。→ **ロールアウトが香連打を実際に打つことを機械確認**（test_v98で、香連打局面のロールアウト着手列に相方の連続香攻めが現れることをアサート）。ここが甘いとA/Bが「実質何も正しく比較していない」状態になり得る。
3. **香温存との干渉**: 敵香受けの香温存（最後の香を握る, L5280-5311）や、相方香に対する既存 `receiveChoice` 分岐（死蔵回避 L5260等）を新経路が上書きしないこと。→ ゲートは Solver/受けMC の**後段** `receiveChoice` を温存する構造を維持（MCがパス/undefinedなら従来 `receiveChoice` に落ちる）。
4. **回帰**: partner-香受け挙動に依存する既存テストがあれば期待値変化。→ 最小限・正当化必須（挙動変更のため0とは限らないが、変化は香分岐に限局されるべき）。
5. **G/G_B非対称化**: 片側のみ適用すると v92c で確立した対称性が崩れる。→ 同一diffを両エンジンへ機械適用し sha で確認。

## 検証計画

1. **診断採掘（`mine_kyo_v98.js`）**: 800配牌以上で partner-香-pass 局面を反実仮想分類（beneficial/harmful/neutral）。香連打中断/敵手番奪取阻止を区別集計。条件別判別力から最小識別条件を導出（段階0の出力）。**この採掘スキャンを実装前後で比較**（有害転換が減り有益が保たれることの傍証）。
2. **機械証明（`test_v98_kyo.js`・fail-if明示）**:
   - 診断で得た代表局面の **fail-before（v97でパス）→ pass-after（v98で受け）** を標準opts・複数シードで断言。
   - **発火/非発火の機械証明**: 香連打進行中（`kyanRendaInProgress`真）では**非発火＝パス維持**であること、敵手番奪取リスク局面では発火＝受けであること。
   - **ロールアウト健全性**: 香連打局面で `rollout` が相方の連続香攻めを継続することをアサート（リスク2対策）。
3. **A/B（受け入れの主柱）**: source-levelエンジン変種を `build_variants_v98.js` で構築し無条件化（`G_base`=v97 / `G_f1`=ゲートのみ / `G_full`=ゲート＋タイブレーク / `B_base`/`B_full`）、**mixed-engine head-to-head（paired standard error）**で直接対戦。無ガード保証（`G_base+段階1+段階2===G_full` の文字列完全一致）を確認。**ランタイムフラグでのgated A/Bは使用しない（フラグ伝播の罠）**。本命ペアリング＋帰属ペアリングの両方で z を報告。
4. **協力プレイ観点（ゴール整合）**: 勝率中立以上に加え、(a) 診断の harmful転換率が beneficial を上回らない、(b) `computeCoach` 上で相方支援手（香連打を通すパス）の評価が悪化しない、(c) why文言が人間相方に不自然でない（「香連打を乱す不用意な受け」に見えない）ことを確認。
5. **回帰**: 既存34テスト全PASS（期待値更新は最小・各更新を正当化）。DD中核（`ddApply`/`ddUndo`/`ddKey`/`ddSearch`/`ddRootValues`/`memo`）は本変更の対象外（`receive` 分岐限定）のため、**DD経路の決定性は保存されるはず**。これを、v97の等価ハーネス（`tests/ab/load_engine.js`＋`test_v97_dd_equiv.js`）を**v97↔v98 比較へ流用**した dd 等価スイープで確認する（＝v97とv98でDD返り値が完全一致するはず。一致しなければ receive 分岐以外への想定外の混入を疑い停止）。併せて DD中核領域の sha256 が v97 から不変であることを確認。
6. **byte-exact/構文/語彙**: `saveCareerToStorage`/`loadCareerFromStorage` 原本一致（不変）・G/G_B同一diff・正準sha256前後記録・`node --check`（4 scriptブロック）・禁止語彙grep 0件・`CACHE_NAME` v118。

## 受け入れ基準

1. **診断の妥当性**: 段階0で「partner-香-pass が有害」な部分集合の存在をデータで示し、採用条件が beneficial保持を維持したまま harmful保持を有意に絞ることを示す。**存在しなければ「是正不要」と結論しクローズ**（実装しないことも正当な帰結）。
2. **A/B勝率中立以上**: 本命・帰属とも paired SE で**有意悪化でない**（winrate-neutral以上）。
3. **香連打の保護**: 香連打進行中は非発火＝パス維持であることを機械証明。ロールアウトが香連打を継続することを機械確認（検証したつもりの回避）。
4. **回帰・規律**: 既存34テスト全PASS（期待値更新は最小・正当化）。v97↔v98のdd等価スイープPASS（DD中核不変の確認）。byte-exact規律・`CACHE_NAME` v118・buildコメント（正準sha256前後）。
5. **A/B設計の明記**: source-level変種の無条件化で実施し gated flag を使っていないこと、無ガード保証（文字列一致）をレポートに明記。

## リスクと停止条件（フェーズ4への申し送り含む）

- **段階0で有益がほぼ無い/有害を絞れない** → 実装せずクローズし「香版は是正不要」と記録（v96 F-1'' の再現＝もっともらしい閾値のA/B悪化、を未然に回避）。
- **A/Bで有意悪化** → 撤回し Reviewer 裁定（条件の再設計 or クローズ）。
- **ロールアウトが香連打を打たない**ことが判明 → タイブレーク（段階2）は成立しないため、ゲート拡張（段階1）のみに縮小 or クローズを検討。
- フェーズ3→4は最大3往復目安。収束しなければ自動ループを止め、計画自体（診断設計の見直し要否）を Sonar に確認する。
